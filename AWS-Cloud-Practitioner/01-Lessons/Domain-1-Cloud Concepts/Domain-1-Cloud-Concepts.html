Introduction

What is "the cloud"?

Before cloud computing, running a website meant buying servers, finding space for them, paying for electricity, and hiring someone to maintain them 24/7. Cloud computing changes that model entirely.

The core idea

Instead of owning computing infrastructure, you rent it from AWS — paying for what you actually use, scaling up when you need more, and scaling down when you don't.

The analogy that makes it click: you don't build a power plant to get electricity. You plug in, use what you need, and pay at the end of the month. AWS is the power plant for computing.

Key terms

Data Center
A building full of servers. AWS has thousands worldwide.
Region
A geographic area (US East, Europe, Asia Pacific) with AWS data centers. 30+ globally.
Availability Zone (AZ)
A cluster of data centers within a region, physically separated for fault tolerance. Each region has 3+ AZs.
On-premises
Keeping IT infrastructure in your own building. The traditional approach.
CapEx
Capital Expenditure — buying hardware/software upfront. A large one-time cost.
OpEx
Operating Expenditure — paying monthly for services. Cloud bills are OpEx.
Task Statement 1.1

Benefits of the AWS Cloud

Four core problems AWS solves — and the benefits that come from solving them.

The core value proposition

AWS's value proposition comes down to four problems it solves: huge upfront costs, long wait times to get infrastructure, guessing future capacity needs, and excessive technical complexity. Without AWS, a startup wanting to launch an app either spends $50,000 on servers before knowing if the idea will work — or crashes when it goes viral. With AWS: start at $20/month, scale automatically to millions of users if needed, pay nothing extra if it fails.

🌍

Global Infrastructure

AWS has data centers worldwide. Distance equals delay — serving users from a nearby region cuts load times from seconds to milliseconds. A European company can launch in Australia in 30 minutes instead of 6 months.

⚡

High Availability

Applications spread across multiple Availability Zones keep working even when individual data centers fail. Automatic failover, no manual intervention. Most AWS services target 99.99% uptime — that's 53 minutes of downtime per year.

📈

Elasticity

Computing resources automatically grow when demand spikes and shrink when it drops. A news site on 1 server normally can auto-scale to 50 servers during breaking news — then scale back. Pay for the surge only while it lasts.

🚀

Agility

Test a new idea in 10 minutes for $100 instead of 6 months for $50,000. If it works, scale it up. If it doesn't, shut it down. Low cost of experimentation means more experiments, more learning, more wins.

💡

Exam tip: Matching scenarios to benefits

Serve users globally with fast performance → Global Infrastructure. Stay up despite data center failures → High Availability / Multiple AZs. Handle traffic spikes automatically → Elasticity. Test ideas quickly without large investment → Agility.

Task Statement 1.2

The Well-Architected Framework

AWS studied why cloud projects fail and built a framework to prevent those failures. Six pillars for building systems that work as intended.

What the framework is for

Systems get hacked when security isn't built in from the start. Costs explode when nobody monitors what's running. Systems crash when they aren't designed for failure. The Well-Architected Framework is a set of design principles to avoid these patterns — like a building code for cloud architecture.

1
Operational Excellence
"Run smoothly, improve constantly"

Monitor everything so you know about problems before users do. Automate operations so routine tasks happen without human intervention. Learn from every failure to prevent recurrence.

2
Security
"Protect everything, always"

Defense in depth: layer multiple security controls so if one fails, others protect you. Principle of least privilege: give users only the minimum access they need. Encrypt data at rest and in transit. Log everything with CloudTrail.

3
Reliability
"Keep working when things break"

Design for failure — assume every component will eventually fail and build around it. Deploy across multiple AZs. Automate recovery so failed components are replaced in seconds, not hours. Test your disaster recovery regularly.

4
Performance Efficiency
"Use resources effectively"

Right-size resources to match actual workload. Use managed services (RDS, DynamoDB) instead of self-managing. Cache frequently read data with ElastiCache. Use CloudFront CDN to serve content from the location nearest to each user.

5
Cost Optimization
"Spend wisely, eliminate waste"

Pay only for what you use — shut down idle resources. Match pricing models to workloads (On-Demand, Reserved, Spot). Use AWS Cost Explorer and Compute Optimizer to find savings. Review monthly — costs drift without vigilance.

6
Sustainability
"Minimize environmental impact"

Run only what you need — idle resources waste energy. Prefer managed and serverless services (shared infrastructure is more efficient). Choose regions with higher renewable energy where latency isn't a constraint.

📝

Exam: Matching problems to pillars

System crashes under load → Reliability. Slow response times → Performance Efficiency. Unexpectedly high AWS bill → Cost Optimization. Data breach → Security. Missed alerts, manual fixes → Operational Excellence. Carbon footprint concerns → Sustainability.

Task Statement 1.3

Migration Strategies

Most companies don't start in the cloud. The question is how to get there — and not every application should be moved the same way.

The AWS Cloud Adoption Framework (CAF)

Migrations fail most often not because of technical problems, but because people, governance, and business alignment weren't addressed. The CAF organizes migration planning into 6 perspectives: Business (why migrate and what ROI?), People (training, role changes, change management), Governance (policies, cost controls, compliance), Platform (which services, what architecture?), Security (encryption, access controls, compliance), and Operations (monitoring, incident response, deployments).

The 6 R's of Migration

Not every application should be migrated the same way. Match the strategy to the application.

Rehost

Lift and Shift

Move as-is, no changes

Move the application to an AWS server exactly as it is. Fastest migration option. Doesn't leverage cloud-native features yet, but gets you off on-premises hardware quickly.

Use when: Fast migration is the priority; limited time for changes; first step before future optimization.
Replatform

Lift, Tinker, and Shift

Minor modifications for cloud benefits

Move the app with small changes to use AWS managed services. Example: switch from a self-managed MySQL server to Amazon RDS. AWS now handles backups, patching, and failover — minimal code changes required.

Use when: You want cloud benefits without a full redesign. The practical middle ground.
Repurchase

Drop and Shop

Replace with SaaS

Replace the existing application with a cloud-native SaaS product. Example: replace on-premises Exchange email with Microsoft 365 — monthly subscription, zero infrastructure to manage.

Use when: Good SaaS alternatives exist; existing software is outdated or expensive to maintain.
Refactor

Re-architect

Redesign for cloud-native

Completely redesign the application to take full advantage of cloud. Break a monolith into microservices, add auto-scaling, use serverless. Highest effort, highest long-term benefit.

Use when: The application is strategically critical and the investment is justified.
Retire

Turn it off

Decommission unused systems

Turn off applications that are no longer needed. Most large organizations have 15–30% of applications that could be retired. Immediate cost savings, zero complexity tradeoff.

Use when: Usage is near zero; functionality replaced by another system.
Retain

Keep on-premises for now

Not ready to migrate yet

Leave the application on-premises. Avoid high-risk migrations with complex hardware dependencies or compliance constraints. Revisit when the timing is right.

Use when: Complex compliance, specialized hardware dependencies, or scheduled for replacement soon.
Task Statement 1.4

Cloud Economics

How cloud computing fundamentally changes the way businesses spend money on technology.

Fixed vs. Variable Costs

Traditional IT (Fixed Costs)

Pay same amount regardless of usage
Buy for worst-case peak demand
Average server utilization: 5–15%
High upfront capital investment (CapEx)
Costs stay fixed when business slows

Cloud Computing (Variable Costs)

Pay only for what you actually use
Scale up when needed, down when not
Pay for exactly the capacity you use
Monthly operational expense (OpEx)
Costs align with business activity

Hidden Costs of On-Premises IT

True Total Cost of Ownership (TCO)

Companies comparing on-premises to cloud often only look at server purchase price vs. cloud monthly fees. This is misleading. On-premises IT has significant hidden costs: facilities (data center floor space, electricity, cooling, physical security), operations (maintenance contracts, backup systems, disaster recovery testing), labor overhead (employee benefits at 30–40% on top of salary, training, overtime during outages, recruiting costs), compliance (annual security audits, certifications, incident response), and opportunity costs (IT staff time spent on maintenance instead of building products).

Studies consistently show the real TCO for on-premises IT is 2–3x higher than the visible hardware and software costs. AWS pricing includes all of these categories transparently in one bill.

Key Economic Concepts

📋

Licensing Strategies

BYOL (Bring Your Own License): Use existing software licenses on AWS infrastructure. Pay AWS only for servers. Good when you have licenses with time remaining — no reason to pay twice.

Included Licensing: License cost bundled into the AWS service price. Simpler management, AWS handles compliance, always on a supported version. Higher ongoing cost but zero licensing overhead.

📐

Rightsizing

Match resource size to actual workload requirements — not too big (wasteful), not too small (performance problems). In the cloud: start conservative, monitor real usage with CloudWatch, then adjust. AWS Compute Optimizer analyzes your actual utilization and recommends the right instance type. Switching from an oversized to a right-sized instance can cut 40–50% of that instance's cost with no performance impact.

🤖

Benefits of Automation

Automation reduces labor costs (provisioning takes minutes, not 8 hours), reduces errors (consistent automated deployments vs. human configuration mistakes), and enables cost control (schedule dev servers to shut down nights and weekends — 60%+ savings on those environments). When traffic spikes, automated scaling responds in 2 minutes; manual response takes 30–60 minutes and users experience the outage.

📊

Economies of Scale

AWS buys millions of servers per year — 40–60% lower hardware costs than any individual company. AWS data centers achieve PUE of ~1.2 vs. typical corporate 2.0+. Thousands of specialists spread across millions of customers. 60–80% server utilization vs. 5–15% for individual companies. All of this translates to lower prices — AWS has cut pricing 100+ times since 2006.

Exam Preparation

Practice Questions

Covering all four task statements in Domain 1. For each question, identify the answer before checking the explanation.

Q1. A startup wants to test a new mobile app idea but doesn't know if it will be successful. What is the primary benefit of using AWS in this scenario?
A AWS guarantees the app will succeed
B AWS allows starting with minimal upfront investment and shutting down cheaply if unsuccessful
C AWS provides the cheapest possible hosting for any workload
D AWS automatically builds the mobile app infrastructure
Answer: B — Low upfront costs change the risk calculation for new ideas. Testing an idea on AWS costs $100, not $100,000. This is the agility benefit of the AWS cloud.
Q2. A company's application crashes every time traffic increases, causing users to lose their work. Which Well-Architected pillar should they focus on?
A Cost Optimization
B Security
C Reliability
D Sustainability
Answer: C — Reliability addresses systems that fail to keep working under varying conditions. The fix involves designing for failure: multiple AZs, auto-scaling, and health checks that automatically replace failed components.
Q3. A company has a 10-year-old CRM system that works well but requires significant maintenance. They want to reduce IT overhead. Which migration strategy is most appropriate?
A Rehost — move it to AWS without changes
B Repurchase — replace it with a cloud-based SaaS CRM
C Refactor — completely redesign it as a cloud-native application
D Retain — keep it on-premises permanently
Answer: B — Repurchase replaces the aging, high-maintenance system with a managed SaaS CRM (like Salesforce). All maintenance becomes the vendor's responsibility. Refactor would be overkill for a CRM with no competitive differentiation.
Q4. A company currently pays $100,000/year for servers regardless of how much they use them. They want costs to align with actual business activity. Which cloud economic concept addresses this?
A Economies of scale
B Variable cost model replacing fixed costs
C BYOL licensing strategy
D Rightsizing
Answer: B — Moving from fixed costs (same regardless of usage) to variable costs (pay for actual usage) means the IT bill rises and falls with the business. Slow months cost less. Busy months cost more — but the business is also making more money.
Q5. According to the Security pillar, what is the best approach for granting employees access to company cloud systems?
A Give all employees full access to prevent bottlenecks
B Grant only the minimum access each employee needs for their specific role
C Only senior employees should have system access
D Require a new IT request every time an employee needs access
Answer: B — The principle of least privilege limits the blast radius of any compromised account. If a marketing employee's credentials are stolen, the attacker can only reach marketing data — not financial systems or customer databases.
Q6. Why can AWS offer database services at a lower cost than most companies could achieve running their own database servers?
A AWS uses lower-quality hardware to reduce costs
B AWS charges less than cost to grow market share
C AWS achieves economies of scale through volume purchasing, shared infrastructure, and operational efficiency
D AWS managed databases have fewer features than self-managed ones
Answer: C — AWS buys hardware at 40–60% discounts individual companies can't get, runs servers at 60–80% utilization vs. 5–15% for individual companies, and spreads thousands of specialists across millions of customers. Every one of these advantages drives down unit cost.
Q7. A migration planning team is identifying what new skills employees need and designing training programs. Which AWS CAF perspective does this represent?
A Business
B People
C Governance
D Operations
Answer: B — The People perspective addresses employee skills, role changes, training programs, and organizational change management. It's specifically about the human side of the migration, not the business case (Business), policies (Governance), or day-to-day system management (Operations).
Q8. A company's website normally handles 5,000 users but spikes to 500,000 during a major product launch. The website must handle the launch without crashes. Which AWS benefit is most directly relevant?
A Global infrastructure and speed of deployment
B High availability across multiple Availability Zones
C Elasticity and automatic scaling
D Economies of scale and lower pricing
Answer: C — The scenario describes variable demand with a 100x traffic spike. Elasticity handles this automatically — scaling up during the launch and back down afterward. High availability (B) is about surviving failures, not handling demand spikes.
