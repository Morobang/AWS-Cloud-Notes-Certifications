Introduction

Security is a shared responsibility

Domain 2 is the largest domain on the Cloud Practitioner exam at 30%. It covers who is responsible for what, how to build a secure environment, how to control access, and how to meet compliance requirements.

The core idea

AWS secures the infrastructure — the physical hardware, data centers, networking, and underlying software. You secure what you put on that infrastructure — your data, your applications, your access controls, and your network configuration.

Neither party can fully secure the environment alone. AWS's secure data centers don't help you if your S3 bucket is publicly accessible. Your encryption settings don't help if AWS's hardware gets breached. That's why it's called the shared responsibility model.

Key terms

Authentication
Proving who you are. Username + password is authentication.
Authorization
Determining what you're allowed to do after your identity is confirmed. IAM policies control authorization.
Encryption at rest
Data is encrypted when stored — in S3, RDS, EBS, etc. Must be enabled explicitly.
Encryption in transit
Data is encrypted while traveling over a network. HTTPS/TLS provides this.
MFA
Multi-Factor Authentication — requires password plus a second factor (phone, hardware token).
Principle of least privilege
Give users only the minimum permissions they need. Nothing more.
Task 2.1

The Shared Responsibility Model

AWS and the customer divide security responsibilities. Understanding where each party's job ends and the other begins is foundational to everything else in this domain.

AWS is responsible for
Security OF the Cloud
Physical data center security — guards, cameras, biometric access
Server hardware and networking gear
Hypervisor and virtualization layer
Global fiber network connecting AWS regions
Host OS patching for managed services (RDS, Lambda)
You are responsible for
Security IN the Cloud
IAM — users, permissions, MFA, credential rotation
Data encryption (at rest and in transit)
Network configuration — security groups, VPC, subnets
Application security — your code's vulnerabilities
Compliance configuration — your workload must meet the standard, not just AWS
⚠

The one constant across every service type

Regardless of whether you're using EC2, RDS, S3, or Lambda — data protection and access management (IAM) are always your responsibility. AWS never manages your encryption settings or your access controls for you.

Responsibility shifts by service type

The more AWS manages, the more AWS takes on. The more control you have, the more you're responsible for.

Security task EC2 (IaaS) RDS (PaaS) S3 Lambda
Physical data center security AWS AWS AWS AWS
Hypervisor / virtualization AWS AWS AWS AWS
Host OS patching You AWS AWS AWS
Database / runtime patching You AWS N/A AWS
Network firewall settings You You You AWS
Data encryption You You You You
IAM and access control You You You You
Application security You You N/A You
ⓘ

Exam decision rule

When a question asks who is responsible: Can the customer configure or change it? If yes → customer's responsibility. Is it about physical infrastructure or AWS's underlying software? If yes → AWS's responsibility. The rule of thumb: if you can SSH into it, patch it, or configure it — it's yours.

Task 2.2

Security, Governance & Compliance

How to protect your data, control your network, audit your account, enforce policies across multiple accounts, and meet regulatory requirements.

Data protection and encryption

Encryption at rest

  • Data encrypted while stored in S3, EBS, RDS, etc.
  • If a hard drive is physically stolen, data is unreadable
  • Must be explicitly enabled per service
  • Keys managed by AWS KMS

Encryption in transit

  • Data encrypted while moving over the network
  • HTTPS/TLS is the standard mechanism
  • AWS services communicate over encrypted connections
  • Enforced by configuring HTTPS endpoints

AWS-managed keys (KMS)

  • AWS creates and rotates keys on your behalf
  • You control who can use them
  • Simpler to manage; meets most compliance needs
  • Every key use is logged in CloudTrail

Customer-managed keys (KMS)

  • You create, configure, and manage the key
  • You control rotation schedule and access
  • Required for regulations that mandate customer key ownership
  • More control, more responsibility

Network security

Foundation

VPC — Virtual Private Cloud

Your private, isolated section of the AWS network. You control IP ranges, traffic routing, and internet access. Resources in private subnets have no direct internet route — databases live here. Resources in public subnets can reach the internet.

Instance-level firewall

Security Groups

Virtual firewalls attached to individual resources. Stateful — allowing inbound traffic automatically allows the response outbound. Default deny: traffic not explicitly allowed is blocked. Write rules for inbound only.

Subnet-level firewall

Network ACLs (NACLs)

Additional firewall layer at the subnet level. Stateless — you must allow both inbound AND outbound traffic explicitly. Processed in numbered order; first match wins. Adds defense-in-depth alongside Security Groups.

Application firewall

AWS WAF

Protects web applications from SQL injection, cross-site scripting (XSS), bot traffic, and known attack patterns. Attaches to CloudFront, Application Load Balancers, or API Gateway. AWS Managed Rules provide pre-built protection.

Logging and monitoring

📄
Audit trail

AWS CloudTrail

Records every API call made in your account — who, what, when, where, how, and whether it succeeded. The audit log for your entire account. Essential for security investigations and compliance audits.

📊
Operational monitoring

Amazon CloudWatch

Collects metrics (CPU, network, disk I/O), logs (application output), and alarms (notify or auto-respond when thresholds are crossed). Dashboards give a live view of system health.

⚙
Configuration tracking

AWS Config

Tracks the configuration state of your resources over time. Can show you what a Security Group looked like at any past date. Config Rules continuously check whether your setup meets your compliance requirements.

Governance

Multi-account management

AWS Organizations

Centrally manage multiple AWS accounts. Enables consolidated billing (one invoice), central policy enforcement, and account isolation. Accounts are grouped into Organizational Units (OUs) that share policies.

Account guardrails

Service Control Policies (SCPs)

Set the maximum permissions available in accounts within your organization. SCPs prevent actions — they don't grant them. An SCP saying "deny resource creation outside us-east-1" blocks all accounts in that OU, including admins.

Resource organization

Resource Tagging

Key-value labels on AWS resources (Environment: Production, Department: Engineering). Enable cost allocation reports, automation scripts, access control by tag, and Config rules that flag untagged resources.

Compliance standards

GDPR

EU Data Protection

Governs personal data of EU residents. Requires consent, right to access/delete data, 72-hour breach notification. Applies to any company handling EU resident data.

HIPAA

US Healthcare

Protects patient health information. Requires encryption, access controls, audit logging, and a Business Associate Agreement (BAA) with AWS.

PCI DSS

Payment Cards

Required for companies processing payment card data. Requires network segmentation, encryption, access controls, and regular vulnerability scanning.

SOC 1/2/3

Audit Reports

Independent audits of internal controls. SOC 2 covers security, availability, confidentiality. Commonly required in enterprise B2B contracts.

ISO 27001

Info Security Mgmt

International standard for information security management systems. Covers policies, controls, and ongoing improvement. Requires independent certification.

AWS Artifact

Compliance Reports Portal

Self-service portal to download AWS's SOC reports, ISO certifications, PCI attestations, and sign the HIPAA BAA — on demand, no sales contact required.

❗

AWS being compliant does not make your workload compliant

AWS holds HIPAA eligibility, PCI DSS certification, SOC 2 certification, and more — but those cover AWS's infrastructure. Your application running on that infrastructure must be separately configured to meet the same requirements: encryption enabled, access controls set, audit logging configured, BAA signed.

Task 2.3

AWS Access Management (IAM)

Identity and Access Management controls who can do what in your AWS account. Everything from creating a user to granting an EC2 instance permission to read S3 goes through IAM.

Authentication — "Who are you?"

  • Proving your identity to AWS
  • Username + password for console
  • Access keys for CLI/API
  • MFA adds a second factor
  • Handled at login time

Authorization — "What can you do?"

  • Determining what actions are permitted
  • IAM policies define allowed/denied actions
  • Checked on every API call
  • Default: deny everything not explicitly allowed
  • Explicit Deny always wins over Allow

The four IAM building blocks

1
Users

Identities for individual people or applications. Each user has credentials (password for console, or access keys for API). One person = one user. Never share credentials between people.

2
Groups

Collections of users with shared permissions. Attach a policy to the group; all members inherit it. Add/remove users from the group to manage access at scale. Groups cannot contain other groups.

3
Roles

Identities that issue temporary credentials when assumed. Used by EC2 instances needing S3 access, Lambda functions, cross-account users, and federated corporate identities. No long-term credentials to manage.

4
Policies

JSON documents defining allowed/denied actions on specific resources. Attached to users, groups, or roles. AWS provides pre-built managed policies; you can also write custom ones for precise control.

The root user

The root user has unrestricted access — protect it carefully

Created when you open your AWS account; tied to the account email address. No IAM policy can restrict the root user. The root user can close the account, change billing information, and modify the support plan — actions IAM users cannot perform.

Best practice: Enable MFA immediately on the root user. Create an IAM administrator for daily work. Never create access keys for root. Never use root for routine tasks.

Multi-Factor Authentication (MFA)

The three factors

  • Something you know — password, PIN
  • Something you have — phone app, hardware token
  • Something you are — fingerprint, face ID
MFA combines at least two factors. Even if a password is stolen, the attacker cannot log in without the second factor.

AWS MFA options

  • Virtual MFA — authenticator app (most common)
  • Hardware MFA — physical device like YubiKey (most secure)
  • SMS MFA — text message codes (least secure, not recommended)
Always enable MFA on the root user and any user with administrative access.

IAM best practices

Practice Why it matters
Enable MFA on the root user Root cannot be restricted by IAM. Compromise is catastrophic — full account control.
Don't use root for daily work Creates unnecessary risk. An IAM admin can do almost everything with a much smaller blast radius.
Create individual users per person Maintains audit trail. Shared accounts make it impossible to know who took an action.
Use groups to manage permissions Changing group permissions affects all members instantly. Simpler than updating per-user policies.
Use roles for applications and services Temporary credentials auto-rotate. No access keys sitting in config files that could be leaked.
Apply principle of least privilege Limits damage from compromised credentials or accidental actions. Start minimal, add as needed.
Rotate access keys regularly Limits the exposure window if a key is compromised. 90-day maximum recommended.
Task 2.4

Security Services & Resources

AWS provides specialized security services for threat detection, vulnerability assessment, DDoS protection, secrets management, and compliance evidence. Know what each service does and when to use it.

Threat detection and monitoring

🔎
Behavioral threat detection

Amazon GuardDuty

Analyzes CloudTrail logs, VPC Flow Logs, and DNS logs to detect threats: compromised instances communicating with malware servers, unusual API calls, port scanning, credential theft attempts. Produces severity-rated findings. Detects but does not block.

📋
Vulnerability assessment

Amazon Inspector

Runs automated security assessments to find software vulnerabilities (CVEs) in EC2 instances, Lambda functions, and container images. Proactively finds weaknesses before attackers do. Provides CVSS severity scores and remediation guidance.

📂
Sensitive data discovery

Amazon Macie

Uses machine learning to scan S3 buckets and identify sensitive data — credit card numbers, social security numbers, names, email addresses (PII). Flags buckets where sensitive data is publicly accessible or poorly protected.

DDoS and web attack protection

🛡
Always-on DDoS protection

AWS Shield Standard

Automatically enabled for all AWS customers at no cost. Protects against common volumetric and protocol DDoS attacks. Most applications are adequately protected by Shield Standard with no configuration required.

🛡
Enhanced DDoS protection

AWS Shield Advanced

Paid service for high-value applications. Adds near real-time attack visibility, 24/7 AWS DDoS Response Team (DRT) access, and cost protection — AWS absorbs scaling costs during attacks. For applications where every minute of downtime is costly.

🔨
Web application firewall

AWS WAF

Filters HTTP requests against rules you define. Blocks SQL injection, XSS, bad bots, and rate-abuse. AWS Managed Rules provide pre-built protection. Attaches to CloudFront, Application Load Balancers, and API Gateway.

📋
Centralized findings

AWS Security Hub

Aggregates findings from GuardDuty, Inspector, Macie, and IAM Access Analyzer into one dashboard. Normalizes findings to a standard format. Also runs automated compliance checks against CIS benchmarks and AWS security best practices.

Secrets and key management

Application credentials

AWS Secrets Manager

Stores database passwords, API keys, and other application secrets securely. Applications retrieve secrets at runtime via API — no hardcoded credentials. Automatic rotation changes passwords on a schedule (e.g., every 30 days) without application downtime.

Encryption key management

AWS KMS

Creates and manages encryption keys used by S3, EBS, RDS, and other services. You control who can use each key and for what. Every key use is logged in CloudTrail. Supports AWS-managed keys (simple) and customer-managed keys (full control).

Compliance resources

Compliance evidence portal

AWS Artifact

Self-service portal to download AWS's compliance reports on demand — SOC 1/2/3, PCI DSS attestations, ISO certifications, HIPAA BAA, GDPR addendums. No sales contact needed. When auditors ask for proof, go here.

Account health advisor

AWS Trusted Advisor

Analyzes your account against best practices in five categories: Cost Optimization, Performance, Security, Fault Tolerance, and Service Limits. Security checks include: public S3 buckets, open Security Groups on sensitive ports, users without MFA, root access keys.

Quick service reference

Security need AWS service
Detect threats from compromised accounts or instances Amazon GuardDuty
Find software vulnerabilities in EC2, Lambda, containers Amazon Inspector
Discover sensitive data (PII) in S3 buckets Amazon Macie
Basic DDoS protection (automatic, free) AWS Shield Standard
Advanced DDoS protection with DRT access and cost coverage AWS Shield Advanced
Filter malicious web requests (SQL injection, XSS, bots) AWS WAF
Centralize all security findings in one view AWS Security Hub
Store and auto-rotate application secrets and passwords AWS Secrets Manager
Manage encryption keys for data at rest AWS KMS
Download AWS compliance certifications and reports AWS Artifact
Get security best practice recommendations for your account AWS Trusted Advisor
Exam Prep

Practice Questions

These questions cover the most commonly tested concepts across all four Domain 2 task statements.

Q1. An EC2 instance is running an application that requires patching of the operating system. The company's security team wants to know who is responsible for applying these patches.
A AWS, because they manage EC2 infrastructure
B The customer, because OS patching on EC2 is the customer's responsibility
C Both equally, since EC2 is a shared service
D AWS, because they have automatic patching tools
Answer: B. EC2 is IaaS. AWS manages the hypervisor and physical hardware. The customer is responsible for the guest operating system, including security patches. Compare this to RDS — AWS patches the database engine because it's a managed service. The rule: if you can SSH into it, patching is your job.
Q2. A company needs to ensure that no account in their AWS Organization can create resources outside of the us-east-1 and us-west-2 regions. What is the most effective mechanism?
A Apply IAM deny policies to each individual user in each account
B Apply a Service Control Policy to the Organizational Unit restricting resource creation to those regions
C Configure VPCs in only those regions
D Enable AWS Config rules that alert on out-of-region resources
Answer: B. SCPs applied to an OU affect all accounts in that OU without requiring changes to individual users or accounts. They enforce guardrails that even account administrators cannot override. Option A requires constant maintenance across many users. Options C and D don't prevent creation — they just limit networking or detect after the fact.
Q3. An application running on EC2 needs to read objects from an S3 bucket. What is the most secure approach?
A Create an IAM user, generate access keys, and hardcode them in the application
B Create an IAM user, generate access keys, and store them in an environment variable on the instance
C Create an IAM role with S3 read permissions and attach it to the EC2 instance
D Make the S3 bucket public so the application can access it without credentials
Answer: C. IAM roles provide temporary, automatically rotating credentials to the EC2 instance. No credentials need to be stored anywhere — not in code, not in environment variables, not in config files. Options A and B use long-term credentials that could be leaked. Option D removes authentication entirely — dangerous for any non-public data.
Q4. A security team is investigating unusual activity in their AWS account from last week. They want to see a complete list of every API call made, who made it, and from which IP address. Which service provides this?
A Amazon CloudWatch
B AWS Config
C AWS CloudTrail
D Amazon GuardDuty
Answer: C. CloudTrail records every API call with identity, timestamp, source IP, and outcome. It's the account audit log. CloudWatch monitors performance metrics and application logs. Config tracks resource configuration history. GuardDuty analyzes those same logs to detect threats — but CloudTrail is the source of truth for who did what.
Q5. A company uses RDS for their database. A developer claims that because RDS is a managed service, the company has no security responsibilities for the database. What is wrong with this claim?
A Nothing — AWS handles all security for managed services
B AWS handles OS and engine patching, but the customer is still responsible for database access controls, encryption settings, and what data they store
C AWS handles everything except OS patching for RDS
D RDS is not a managed service
Answer: B. "Managed" means AWS handles operations (OS patching, DB engine updates, automated backups, hardware). The customer retains responsibility for who can connect to the database, whether data is encrypted, what data is stored, and database user accounts and permissions. Managed reduces operational burden — it doesn't eliminate security responsibility.
Q6. An e-commerce platform experienced a DDoS attack that overwhelmed their servers for 30 minutes. They want enhanced protection that includes 24/7 AWS incident response support and financial protection against scaling costs during attacks. Which service meets these requirements?
A AWS Shield Standard
B AWS Shield Advanced
C AWS WAF
D Amazon GuardDuty
Answer: B. Shield Advanced adds the DDoS Response Team (24/7 support), near real-time attack visibility, and cost protection (AWS absorbs scaling charges during attacks). Shield Standard is already running but doesn't include these features. WAF filters application-layer requests but doesn't handle volumetric network floods. GuardDuty detects threats but provides no DDoS defense.
Q7. A company wants to download a copy of AWS's SOC 2 Type II report to share with a customer conducting a vendor security assessment. Where do they find this document?
A AWS Trusted Advisor
B AWS Security Hub
C AWS Artifact
D AWS Config
Answer: C. AWS Artifact is the self-service portal for accessing AWS compliance documentation — SOC reports, ISO certificates, PCI DSS attestations, and more. It's available 24/7 after accepting a non-disclosure agreement. Trusted Advisor provides best-practice recommendations for your account. Security Hub aggregates security findings. Config tracks configuration changes.
Q8. A healthcare company stores patient records in S3. A compliance officer says "We're HIPAA compliant because we're using AWS, which is HIPAA-eligible." What is incorrect about this statement?
A AWS is not HIPAA-eligible
B AWS HIPAA eligibility covers the infrastructure; the company must separately configure encryption, access controls, audit logging, and sign a Business Associate Agreement
C HIPAA compliance is automatic when data is stored in S3
D The statement is correct — AWS eligibility covers all customer workloads
Answer: B. This is the shared responsibility model applied to compliance. AWS's HIPAA eligibility means their infrastructure meets HIPAA requirements. The customer must still enable encryption for PHI, implement access controls, configure audit logging (CloudTrail), and execute a HIPAA Business Associate Agreement with AWS. Being on HIPAA-eligible infrastructure is necessary but not sufficient.
