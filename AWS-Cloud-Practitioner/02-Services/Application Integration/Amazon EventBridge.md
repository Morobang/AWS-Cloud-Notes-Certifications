# Amazon EventBridge

---

## What is Amazon EventBridge?

Amazon EventBridge is a **serverless event bus** that routes events from sources to targets based on rules. Applications publish events; EventBridge evaluates each event against your rules and delivers matching events to the appropriate targets.

The key differentiator from SNS is that EventBridge **understands event content** — it can filter and route based on the data *inside* the event, not just which topic it was published to.

---

## How it works

**Events** are JSON documents that describe something that happened:
```json
{
  "source": "aws.ec2",
  "detail-type": "EC2 Instance State-change Notification",
  "detail": { "state": "stopped", "instance-id": "i-1234567890" }
}
```

**Rules** match events based on patterns (event source, detail-type, specific field values) and route matching events to one or more **targets**.

**Targets** can be: Lambda functions, SQS queues, SNS topics, Step Functions, Kinesis streams, EC2 instances, API destinations (HTTP endpoints), and more.

---

## Event sources

- **AWS services**: EC2 state changes, S3 object creation, RDS events, CodePipeline state changes, and hundreds more
- **SaaS applications**: Salesforce, Zendesk, Datadog, PagerDuty, and others through EventBridge partner integrations
- **Your own applications**: Any application can publish custom events using the EventBridge API

---

## When to use it

**React to AWS service events** — When an EC2 instance is stopped, automatically send a Slack notification via Lambda.

**SaaS event routing** — When a Salesforce opportunity is won, trigger an order provisioning workflow in your system.

**Event-driven microservices** — Different services publish events to EventBridge; other services subscribe to the events they care about, with no direct coupling between them.

**Scheduled events** — EventBridge also supports cron-like scheduled rules to trigger targets on a time-based schedule (like CloudWatch Events).

---

## EventBridge vs SNS

| | Amazon EventBridge | Amazon SNS |
|-|-------------------|------------|
| Routing | Content-based (filter on event fields) | Topic-based (all subscribers get all messages) |
| Sources | AWS services, SaaS, custom | Any publisher |
| Filtering | Rich filtering on any JSON field | Basic message attribute filtering |
| Use case | Event-driven architectures with routing | Fan-out to multiple subscribers |
| Best for | When different events need different targets | When all consumers need the same message |

---

## Exam focus

- EventBridge = **serverless event bus** with **content-based routing**
- Routes events based on what's *inside* the event (not just who sent it)
- Integrates with AWS services, SaaS partners, and custom applications
- Supports **scheduled rules** (replaces old CloudWatch Events)
- Key differentiator from SNS: EventBridge routes by content; SNS delivers to all subscribers

## Practice questions

**Q1.** A company wants to automatically trigger different workflows based on the type of events generated by their AWS services. When an RDS instance fails over, they want to notify the on-call team; when an EC2 instance is terminated, they want to trigger a cleanup Lambda. Which service enables this content-based routing?

A) Amazon SNS  
B) Amazon SQS  
C) Amazon EventBridge  
D) AWS Step Functions

**Answer: C** — EventBridge routes events to different targets based on the content of the event (detail-type, source, specific field values). This content-based routing is what differentiates it from SNS, which delivers all messages to all subscribers of a topic regardless of content.
