Introduction

Keeping Data Pipelines Running

Domain 3 covers the operational side of data engineering: automation, analytics, and ongoing maintenance of production data systems.

Operations is the hardest part of data engineering

Building a pipeline once is straightforward. Running it reliably at scale — with automated deployments, proper monitoring, cost controls, and fast incident response — is where most engineering effort goes. Domain 3 tests your ability to automate data processing (CI/CD, IaC, event-driven triggers), query and visualize data (Athena, QuickSight, Flink), and maintain production pipelines (CloudWatch, DLQs, cost optimization, data quality).

Key concepts to master

Event-driven automation
S3 events, EventBridge schedules/rules, and SNS/SQS trigger pipeline steps without polling.
IteratorAge
Most important Kinesis metric: age of oldest unprocessed record. Growing IteratorAge = consumer can't keep up.
Dead Letter Queue (DLQ)
SQS queue or SNS topic that receives messages that fail after max retry attempts. Prevents data loss.
Glue Job Bookmarks
Track which S3 files/DynamoDB records have been processed. Re-running a job only processes new data.
SPICE
QuickSight's in-memory engine. Import data for sub-second dashboard performance. Separate from direct query.
Tumbling window
Fixed-size, non-overlapping time window in streaming. Every record belongs to exactly one window.
Watermark
Flink mechanism for handling late-arriving records. Defines how long to wait before closing a window.
Workgroup (Athena)
Isolate users/teams, enforce scan limits (cost control), and separate query results per team.
Task Statement 3.1

Automate Data Processing

CI/CD pipelines for data, infrastructure as code, event-driven triggers, and compute options for batch jobs.

Data pipelines need the same rigor as application code

Data pipelines break silently — a schema change upstream doesn't throw an exception; it produces corrupted output. Treating pipeline code as first-class software (version control, automated testing, IaC deployment) is the foundation of operational excellence in data engineering.

Event-Driven Triggers

📅

EventBridge Scheduler

Cron or rate-based scheduling for pipelines. Replaces Lambda cron functions. Can invoke Step Functions, Lambda, Glue, Batch, or any API call as a target. Supports one-time scheduled events. Cost: per schedule invocation.

🪣

S3 Event Notifications

S3 triggers on object creation, deletion, or restore. Route to Lambda (for immediate processing), SQS (for queued processing), SNS (for fan-out), or EventBridge (for complex routing). Use EventBridge for cross-account routing or Step Functions start.

🔗

Glue Triggers

Within a Glue Workflow: schedule triggers (cron), on-demand triggers (manual), and conditional triggers (start next job when previous job SUCCEEDED). Glue Workflows orchestrate Glue-only pipelines; use Step Functions for cross-service orchestration.

📨

DMS and CDC Events

DMS task state changes publish to EventBridge. Build automation: when a full-load task completes, EventBridge triggers Step Functions to start the incremental CDC phase. No polling required.

⚠️

AWS Batch vs Lambda for long-running jobs

Lambda's 15-minute maximum is a hard limit. If a processing job can take hours (e.g., large-file compression, video transcoding, model training), use AWS Batch. Batch runs arbitrary Docker containers on EC2 or Fargate with no time limit. It also supports array jobs — submit one job definition that fans out to hundreds of parallel job attempts, each processing a different input (e.g., one file per day of the year = 365 parallel jobs).

Task Statement 3.2

Analyze Data and Create Visualizations

Interactive SQL queries with Athena, streaming analytics with Managed Flink, and business dashboards with QuickSight.

Amazon Athena — Serverless SQL on S3

💰

Cost Optimization

Athena charges $5 per TB scanned. Use Parquet or ORC (columnar — only scans needed columns), Hive-style partitioning (year=/month=/day= — only scans matching partitions), and ZSTD compression. Together: 90%+ cost reduction vs raw CSV/JSON.

🏢

Workgroups

Separate query contexts per team or environment. Set per-query scan limits (stop query if it would scan >10GB). Set per-workgroup data usage limits (SNS alert if team scans >1TB/day). Control result locations in S3. Use for cost governance across teams.

📂

CTAS and Views

CREATE TABLE AS SELECT writes query results back to S3 as a new partitioned Parquet table. Use CTAS to pre-aggregate data for fast dashboards. Views are logical (no data stored) — evaluated at query time. Use for access control (grant on view, not base table).

🔗

Federated Query

Lambda-based connectors query data outside S3: RDS, DynamoDB, Redshift, DocumentDB, CloudWatch Logs. Join a DynamoDB enrichment table to S3 data lake in a single Athena SQL query. Results spill to S3 if they exceed memory.

Amazon Managed Flink — Streaming Analytics

When Lambda isn't enough for streaming

Lambda processes Kinesis records independently — it has no memory between invocations, so it can't aggregate "count of events in the last 5 minutes." Amazon Managed Flink (formerly Kinesis Data Analytics for Apache Flink) is the service for stateful streaming: windowed aggregations, stream-to-stream joins, session tracking, and anomaly detection over time. State is checkpointed to S3 for fault tolerance.

Window typeBehaviorExample
TumblingFixed size, non-overlapping. Each event belongs to exactly one window.Count orders per 1-minute window: 12:00–12:01, 12:01–12:02, ...
SlidingFixed size, overlapping. Events can belong to multiple windows.Moving average: sum of last 5 minutes, updated every 1 minute.
SessionDynamic size. Window closes after inactivity gap.User session: group clicks with no more than 30 minutes between events.
💡

Watermarks handle late data

In real-world streams, records arrive out of order (network delays). A watermark tells Flink: "allow up to 10 seconds of late arrival before closing a window." Records arriving after the watermark passes are either dropped or routed to a side output for separate handling. Without watermarks, Flink would either wait forever (correctness) or close windows early (incompleteness).

⚠️

IteratorAgeMilliseconds is the most tested Kinesis metric

On the exam, if a Kinesis consumer is falling behind (records are hours old by the time they're processed), the root cause is almost always one of: (1) not enough shards (throughput bottleneck) — fix: reshard; (2) consumer processing too slowly — fix: add more consumer instances using Enhanced Fan-Out; (3) Lambda reserved concurrency too low — fix: increase. The metric that tells you this is happening is IteratorAgeMilliseconds growing over time.

Dead Letter Queues and Error Handling

📮

Lambda DLQ

For async invocations (S3 events, SNS), Lambda retries twice by default. After 3 attempts, routes to a configured DLQ (SQS) or destination (EventBridge, Lambda, SQS, SNS). Configure DestinationConfig with OnFailure and OnSuccess separately.

📥

SQS + Lambda DLQ

For SQS event source mappings, configure both: an SQS queue DLQ (for messages Lambda never processes) and a Lambda DLQ (for processing errors). SQS visibility timeout should be >6× Lambda timeout to avoid duplicate processing during retries.

🌊

Kinesis Error Handling

Lambda Kinesis ESM retries failed records until they expire (24h or 7 days depending on retention). Use BisectOnFunctionError: true to split a failing batch in half and isolate the bad record. Configure DestinationConfig to route poisoned records to SQS DLQ rather than blocking the entire shard.

🔍

Glue Data Quality

DQDL rules (Data Quality Definition Language) validate data inline during Glue jobs. Rules like IsComplete "customer_id", RowCount > 1000, ColumnValues "amount" between 0 and 1000000. Configure action on failure: FAIL_JOB (stop pipeline), WARN (continue with alert), or QUARANTINE (route bad records to separate S3 path).

Practice Questions

Domain 3 Exam Practice

These questions reflect the style and difficulty of actual DEA-C01 exam questions. The correct answer is highlighted in green.

Q1. A Kinesis Data Streams consumer is processing records from a stream with 4 shards. The CloudWatch metric GetRecords.IteratorAgeMilliseconds has been growing steadily over the past 24 hours and is now at 2 hours. What is the most likely cause and recommended fix?
A The stream's data retention period is too short; increase it to 7 days
B The producer is sending records with incorrect partition keys; redistribute keys evenly
C The consumer cannot process records fast enough; add more consumer instances or split shards
D The CloudWatch metric is a false alarm; IteratorAge fluctuates normally
C is correct. IteratorAgeMilliseconds measures the age of the oldest record a consumer hasn't processed yet. A steadily growing value means the consumer is falling behind — it processes records slower than new records arrive. Fix: add more consumer instances (Enhanced Fan-Out gives each consumer its own read throughput), or split shards if you're also hitting write limits. Retention period doesn't affect processing speed.
Q2. A Glue ETL job processing 500GB of data is failing with an OutOfMemoryError. The job currently uses the G.1X worker type with 10 workers. What is the correct fix?
A Increase the number of workers from 10 to 50
B Change the worker type from G.1X to G.2X
C Enable Glue Auto Scaling to automatically add workers
D Split the input data into smaller files before running the job
B is correct. An OOM error indicates insufficient memory per executor, not insufficient parallelism. G.1X workers have 16GB memory; G.2X workers have 32GB. Doubling the per-worker memory directly addresses the OOM. Adding more workers (A) doesn't help because the OOM is happening within a single executor on one partition. Auto Scaling adds workers, not memory per worker.
Q3. A data engineer needs to monitor data pipeline record counts. When fewer than 1,000 records are processed in a Glue job run, the team should be alerted. Glue doesn't natively publish a "records processed" metric. What is the best approach?
A Parse Glue job logs in CloudWatch Logs and create a metric filter on record count patterns
B Publish a custom CloudWatch metric from within the Glue job using put_metric_data, then create an alarm on it
C Use CloudTrail to capture Glue API calls and extract record counts
D Enable Glue job profiling and monitor the profiling output in S3
B is correct. Custom CloudWatch metrics allow you to publish any business-level metric from application code (boto3 put_metric_data). Add a call at the end of the Glue job to publish the record count, then create a CloudWatch Alarm that triggers when the metric drops below 1,000. CloudTrail tracks API calls, not data-level metrics. Log metric filters work but are more fragile (depends on exact log format).
Q4. A Lambda function processes messages from an SQS queue. Occasionally, malformed messages cause the Lambda function to throw an exception. After several retries, these messages need to be preserved for investigation rather than lost. What should be configured?
A Set Lambda's reserved concurrency to 0 when errors occur
B Configure a Dead Letter Queue on the SQS source queue with maxReceiveCount set to the desired retry count
C Enable Lambda Destinations with an OnFailure target pointing to another Lambda
D Use an SQS FIFO queue instead of a standard queue
B is correct. For SQS → Lambda event source mappings, configure a Dead Letter Queue on the SQS queue (not Lambda), using RedrivePolicy with a maxReceiveCount. After the message is received (and returned to the queue after Lambda failure) that many times, SQS automatically moves it to the DLQ. Lambda Destinations (C) work for async invocations, not SQS ESM. FIFO guarantees ordering, not error handling.
Q5. An Athena workgroup is being used by a data science team that frequently runs expensive queries against a large S3 data lake. The data engineering team needs to prevent any single query from scanning more than 50GB. How should this be configured?
A Create an S3 bucket policy that limits reads to 50GB per request
B Use AWS Cost Explorer to alert when Athena costs exceed a threshold
C Configure a per-query data usage limit of 50GB on the Athena workgroup
D Create an IAM policy that denies Athena queries scanning more than 50GB
C is correct. Athena workgroups support per-query data usage controls: when a query would scan more than the configured limit, Athena cancels it before incurring the full cost. This is configured directly on the workgroup under "Data usage controls." S3 bucket policies can't limit read volume. Cost Explorer is reactive (after the fact). IAM doesn't have a "bytes scanned" condition.
Q6. A real-time pipeline needs to calculate the count of user logins per 5-minute window, updated every minute. Which Amazon Managed Flink window type should be used?
A Tumbling window of 5 minutes
B Sliding window of size 5 minutes and slide of 1 minute
C Session window with a 5-minute gap
D Global window with a periodic trigger every minute
B is correct. A sliding window with size=5 minutes and slide=1 minute produces a new result every minute, but each result covers the last 5 minutes of data. This creates overlapping windows: 00:00–05:00, 00:01–05:01, 00:02–05:02, etc. A tumbling 5-minute window would only produce results at 5-minute boundaries (non-overlapping). A session window closes based on inactivity, not time intervals.
Q7. An EMR cluster's HDFS storage is at 92% utilization. The cluster is running a daily batch job that processes log files. What is the most effective immediate action?
A Increase the master node instance size
B Move intermediate and output data to Amazon S3 using EMRFS instead of HDFS
C Compress all HDFS data using GZIP
D Add more task nodes to the cluster
B is correct. EMRFS (EMR File System) uses S3 as the underlying storage instead of HDFS. Moving data to S3 immediately frees HDFS space and is the standard best practice for EMR — S3 provides durable, unlimited storage at lower cost, while HDFS is typically only used for data that needs low-latency local disk access. Task nodes don't contribute HDFS storage (only core nodes do). Compression helps at the margins but doesn't solve 92% utilization.
Q8. A QuickSight dashboard is used by 500 executives who all load the same sales summary report at the same time every Monday morning. The report queries a large Redshift table and is slow. What is the best way to ensure fast, consistent dashboard performance?
A Scale up the Redshift cluster every Monday morning before 9 AM
B Import the Redshift query results into QuickSight SPICE and schedule a weekly refresh
C Create a Redshift materialized view and grant QuickSight access to it
D Switch to Amazon Athena with the same data in S3 for cheaper queries
B is correct. SPICE (Super-fast, Parallel, In-memory Calculation Engine) imports data once and serves dashboard queries from memory. 500 concurrent users hit SPICE, not Redshift — zero load on the data warehouse. Schedule a weekly SPICE refresh every Sunday night so Monday data is ready. Option C (materialized view) reduces Redshift compute but still requires 500 concurrent Redshift queries. Option A is operationally fragile and doesn't solve concurrent user load.
