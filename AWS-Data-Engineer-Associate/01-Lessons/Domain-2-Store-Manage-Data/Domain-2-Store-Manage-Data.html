Introduction

Data Storage Strategy

Domain 2 covers how to choose, design, optimize, and maintain the data stores at the heart of any analytics platform.

The storage layer is where most design decisions live

Choosing the wrong data store is the most expensive mistake in data engineering. The exam tests your ability to match workload characteristics — query patterns, throughput, latency, consistency, and cost — to the right AWS service. It also tests schema management (Glue Data Catalog, Iceberg, format evolution) and data lifecycle (S3 tiering, retention, archiving).

The key mental model: OLTP vs OLAP vs Object Storage vs Streaming. Once you categorize the workload, most service selections follow naturally.

Key concepts to master

OLTP
Online Transaction Processing. Row-oriented, optimized for many short read/write transactions. RDS, Aurora, DynamoDB.
OLAP
Online Analytical Processing. Column-oriented, optimized for large aggregations over many rows. Redshift, Athena.
Medallion Architecture
Bronze (raw) → Silver (cleaned Parquet) → Gold (business aggregations). Organizes S3 data lake layers.
Distribution Style
Redshift setting controlling how data is spread across nodes: KEY, ALL, EVEN, or AUTO.
Sort Key
Redshift setting controlling physical row order on disk. Compound (range queries) vs Interleaved (multi-column).
Apache Iceberg
Open table format providing ACID transactions, schema evolution, time travel, and row-level deletes on S3.
Single-Table Design
DynamoDB pattern storing multiple entity types in one table using generic PK/SK patterns to minimize API calls.
S3 Intelligent-Tiering
Automatically moves objects between access tiers based on usage patterns. Ideal for unpredictable access.
Task Statement 2.1

Choose a Data Store

Match workload characteristics to the right AWS storage service. The exam frequently tests this decision tree.

The data store decision framework

Start with the access pattern: Is it transactional (row-by-row) or analytical (aggregate scans)? Then consider data structure: relational, key-value, document, graph, time-series, or object? Finally, evaluate scale requirements: throughput, latency, and whether the schema is fixed or flexible.

⚠️

Exam pattern: wrong service for the workload

Many exam questions present a performance problem (high latency, timeouts, query too slow) and ask you to fix it by changing the data store. RDS cannot handle millions of writes/sec → DynamoDB. Redshift cannot do millisecond lookups → DynamoDB or ElastiCache. S3 alone cannot serve fast queries → Redshift Spectrum or Athena.

🗄️

When to choose DynamoDB

Consistent single-digit ms latency regardless of scale. Access patterns are known and fixed. Data fits a key-value or document model. High write throughput (thousands/sec). Consider: game state, user profiles, IoT telemetry, shopping carts.

📊

When to choose Redshift

Complex analytical queries joining large tables. Aggregations over billions of rows. BI tools need fast responses. Data is structured and schema-on-write. Consider: sales reports, financial analytics, executive dashboards.

🪣

When to choose S3 + Athena

Ad-hoc queries on semi-structured data. No pre-provisioned infrastructure. Variable query frequency. Data already in S3. Query cost per scan is acceptable. Consider: log analysis, audit queries, data exploration.

🔍

When to choose OpenSearch

Full-text search across documents. Log aggregation with free-form queries. Near-real-time search latency. Visualizations with OpenSearch Dashboards. Consider: application logs, e-commerce search, security analytics.

Task Statement 2.2

Understand Data Cataloging, Schema Evolution & Optimization

How to design data stores for query performance, handle schema changes over time, and manage metadata across the data lake.

Redshift Performance Optimization

Redshift MPP architecture

Redshift stores data in columnar format across multiple nodes (slices). Data is compressed by column, and queries execute in parallel. Performance depends on how data is physically distributed and sorted — two settings you control at table creation time: distribution style and sort key.

Distribution StyleHow it worksBest for
KEYRows with same key value go to same node. Matching keys between two tables = collocated join (no data movement).Large fact tables joined to large dimension tables on a common key (e.g., customer_id).
ALLFull copy of table on every node. Eliminates data movement for joins.Small, slow-changing dimension tables (country codes, product categories).
EVENRound-robin distribution across all nodes. No join collocation.Tables that are not frequently joined. Avoids skew when no obvious join key.
AUTORedshift chooses ALL for small tables, EVEN for large. Can change over time.Default. Let Redshift decide; review SVCS_TABLE_INFO to verify.
Sort Key TypeBest forExample
Compound (default)Range-based filters on leading column. Effective when WHERE clause matches first sort column.COMPOUND SORTKEY(order_date, region)
InterleavedMultiple independent filter columns with equal selectivity. Slower VACUUM.INTERLEAVED SORTKEY(customer_id, product_id, date)
💡

VACUUM and ANALYZE — always a pair

VACUUM reclaims storage from deleted rows and re-sorts unsorted rows back into sort key order. Run after large DELETE or UPDATE operations. ANALYZE updates query planner statistics — run after VACUUM so the planner knows the new row distribution. Without ANALYZE, the planner may choose suboptimal join strategies. Use VACUUM DELETE ONLY to reclaim space without re-sorting (faster).

DynamoDB Design Patterns

🔑

Single-Table Design

Store multiple entity types in one table. Use generic PK/SK patterns like PK: USER#u001, SK: ORDER#2024-01#o99. Query a user and all their orders with a single API call using begins_with on SK.

📑

GSI — New Access Patterns

Global Secondary Index projects different PK/SK attributes. Use when you need to query by email (not user_id), or by status + date. Max 20 GSIs per table. GSI reads consume capacity from the index, not the base table.

🔀

Write Sharding

If one partition key is too "hot" (millions of writes/sec to same key), append a random suffix: STATUS#PENDING_3. Spread load across multiple partitions. Scatter-gather to query all shards.

⏱️

DynamoDB TTL

Set an attribute as TTL epoch timestamp. DynamoDB auto-deletes expired items within 48 hours at no cost. Use for session expiry, temporary caches, event logs. TTL deletes appear in DynamoDB Streams for downstream processing.

Apache Iceberg — Transactional Data Lake

What Iceberg solves

Traditional S3 data lakes have no ACID guarantees. Two writers can corrupt each other's data. Deleting a row means rewriting entire Parquet files. Iceberg adds a metadata layer on top of S3 files that tracks every change as a snapshot. This enables: ACID transactions, row-level deletes (GDPR erasure), time travel (query data as of yesterday), schema evolution (add/rename/remove columns safely), and partition evolution (change partition strategy without rewriting data).

Iceberg capabilityHow it worksExam scenario
ACID transactionsOptimistic concurrency via snapshot isolation. Writers create new snapshot; conflicts abort.Two Glue jobs writing to same table simultaneously — use Iceberg.
Row-level deletesMerge-on-read: write delete file listing row positions to remove. No full file rewrite required.GDPR "right to erasure" — delete customer rows by ID.
Time travelQuery older snapshots: SELECT * FROM table FOR SYSTEM_TIME AS OF '2024-01-01'Recover from accidental DELETE or audit historical state.
Schema evolutionAdd/rename/drop columns tracked in metadata. Old files still readable by mapping column IDs.Add new source field without rewriting existing Parquet files.
Partition evolutionChange partition spec going forward without rewriting historical data.Switch from monthly to daily partitions as data volume grows.
✅

Iceberg on AWS

Iceberg is natively supported by AWS Glue (Glue Data Catalog stores Iceberg metadata), Amazon Athena (DDL + DML including DELETE/UPDATE), Amazon EMR (PySpark with Iceberg jar), and Amazon S3 Tables (managed Iceberg buckets with automatic compaction). The catalog is the source of truth for all table metadata.

Schema Management — Glue Data Catalog

ParquetAdd new columns (safe). Rename/remove columns breaks readers unless catalog maps them.S3 data lake default. Columnar, compressed. Athena reads by column name, tolerates added columns.

⚠️

Glue mergeSchema option

When reading Parquet files with different schemas (some files have a new column added later), set mergeSchema = true in the Glue job or Spark read options. Without this, Spark uses only the schema from the first file read and silently drops columns that don't match. This is a common exam scenario: "new columns are missing in the output" → mergeSchema.

Task Statement 2.3

Manage the Data Lifecycle

Tiering data to lower-cost storage over time, setting retention policies, archiving, and preventing accidental deletion.

Data ages out of active use

Most data engineering pipelines produce data that is accessed frequently at first (last 30 days), then occasionally (last year), then rarely or never (historical archive). AWS S3 storage classes map directly to this lifecycle: Standard → Standard-IA → Glacier Instant Retrieval → Glacier Flexible Retrieval → Glacier Deep Archive. Lifecycle policies automate these transitions based on object age.

S3 Storage ClassRetrievalMin storageBest for
S3 StandardMillisecondsNoneActive data accessed frequently (>1x/month)
S3 Standard-IAMilliseconds30 daysData accessed less than monthly, must be quickly retrievable
S3 One Zone-IAMilliseconds30 daysReproducible data (can be regenerated if lost). No multi-AZ replication.
S3 Intelligent-TieringMilliseconds (frequent/infrequent) or minutes (archive)NoneUnknown or unpredictable access patterns. Monitors and auto-tiers.
S3 Glacier Instant RetrievalMilliseconds90 daysArchive data accessed ~quarterly. Same latency as Standard but lower cost.
S3 Glacier Flexible Retrieval1–12 hours (Standard), 3–5 hours (Expedited), 12 hours (Bulk free)90 daysCompliance archives, backups accessed rarely
S3 Glacier Deep Archive12–48 hours180 daysLowest cost. Long-term regulatory retention (7+ years). Rarely accessed.
💡

S3 Lifecycle — transition rules in JSON

Lifecycle rules are applied to a bucket or prefix. Common pattern: Standard → Standard-IA after 30 days → Glacier Flexible after 90 days → expire (delete) after 365 days. Rules can also delete incomplete multipart uploads (cost leak if not cleaned up).

# S3 lifecycle rule — Bronze layer auto-tiering { "Rules": [{ "ID": "bronze-lifecycle", "Status": "Enabled", "Filter": { "Prefix": "bronze/" }, "Transitions": [ { "Days": 30, "StorageClass": "STANDARD_IA" }, { "Days": 90, "StorageClass": "GLACIER" }, { "Days": 365, "StorageClass": "DEEP_ARCHIVE" } ], "Expiration": { "Days": 2555 }, // 7 years retention then delete "AbortIncompleteMultipartUpload": { "DaysAfterInitiation": 7 } }] }
🔒

S3 Object Lock (WORM)

Governance mode: protect against accidental deletion; users with special IAM permission can override. Compliance mode: no one (including root) can delete until retention period expires. Use Compliance for SEC, FINRA, HIPAA regulatory requirements. Requires versioning enabled.

🗂️

DynamoDB TTL

Set a Unix epoch timestamp attribute (e.g., expires_at) on items. DynamoDB scans for expired items and deletes them within 48 hours — no cost for TTL deletes. TTL deletions appear in DynamoDB Streams so downstream consumers can react (e.g., invalidate a cache).

📦

Redshift Data Archiving

Use UNLOAD to export old partitions to S3 as Parquet, then delete from Redshift to reclaim storage. Query archived data with Redshift Spectrum without reloading it. For rarely-queried historical data, Spectrum + S3 Glacier Instant Retrieval is significantly cheaper than hot Redshift storage.

🧹

Iceberg Table Maintenance

Iceberg accumulates metadata and small data files over time. Run periodic maintenance: CALL system.expire_snapshots() to remove old snapshots, CALL system.rewrite_data_files() to compact small files, CALL system.remove_orphan_files() to clean unreferenced files. In AWS: Glue Data Catalog handles this for S3 Tables automatically.

Practice Questions

Domain 2 Exam Practice

These questions reflect the style and difficulty of actual DEA-C01 exam questions. The correct answer is highlighted in green.

Q1. A data engineering team needs to store IoT sensor readings from 10,000 devices. Each device sends 100 records per second. Queries always access data by device_id and time range. Which storage solution provides the best performance at this scale?
A Amazon RDS MySQL with a composite index on (device_id, timestamp)
B Amazon DynamoDB with device_id as partition key and timestamp as sort key
C Amazon Redshift with a compound sort key on (device_id, timestamp)
D Amazon S3 with Hive-style partitioning by device_id
B is correct. DynamoDB is designed for high-throughput, single-digit ms access at any scale. The composite key (device_id PK + timestamp SK) perfectly matches the access pattern: get all readings for a device in a time range with a single Query call. RDS cannot handle 1M writes/sec. Redshift is for analytics, not OLTP writes. S3 alone cannot serve fast queries.
Q2. A Redshift data warehouse has a 500-million-row fact table that is always joined to a 200-row country_codes dimension table. Queries are running slowly due to data movement between nodes during joins. What is the most effective fix?
A Change the fact table distribution style to ALL
B Change the fact table distribution style to KEY on country_code
C Change the country_codes dimension table distribution style to ALL
D Add an interleaved sort key to the fact table on country_code
C is correct. Small, slow-changing dimension tables should use ALL distribution — every node gets a full copy, eliminating all data movement during joins. Setting the 500M-row fact table to ALL would create enormous storage overhead. KEY distribution on the fact table would only help if the dimension table were also KEY-distributed on the same key.
Q3. A company has a data lake on S3 where data engineers and data scientists run different PySpark jobs that both write to the same S3 prefix simultaneously. Occasionally, data corruption is observed. Which solution addresses this without changing the storage layer?
A Use S3 Versioning to recover from corruption
B Add an SQS queue between writers to serialize writes
C Convert the S3 prefix to an Apache Iceberg table
D Move the data to Amazon Redshift
C is correct. Apache Iceberg provides ACID transactions on S3 via snapshot isolation. Concurrent writers each create a new snapshot; if two writers conflict, one will fail and retry. S3 Versioning doesn't prevent corruption — it only preserves old versions. SQS serialization is complex and doesn't guarantee consistency. Moving to Redshift changes the storage layer, which the question excludes.
Q4. A data engineer needs to implement a GDPR "right to erasure" capability for customer PII stored in an S3-based data lake. Data is stored in Parquet files, partitioned by date. What is the most cost-effective approach?
A Rewrite all Parquet files after filtering out the customer's rows using an AWS Glue job
B Convert the table to Apache Iceberg and use row-level DELETE to remove the customer's records
C Delete the entire partition containing the customer's data
D Encrypt the customer's data with a unique KMS key and delete the key
B is correct. Iceberg supports row-level DELETE using merge-on-read (delete files), avoiding full Parquet file rewrites. This is targeted, cost-effective, and GDPR-compliant. Option A (rewrite all files) is expensive for large datasets. Option C (delete partition) also deletes other customers' data. Option D (cryptographic erasure) is valid but more complex; Iceberg row-level delete is the standard approach tested on the exam.
Q5. A data pipeline reads Parquet files from S3. Over time, source teams have added new columns to some files. When the Glue ETL job runs, the new columns are missing from the output. What is the root cause and fix?
A The Glue crawler hasn't re-crawled the S3 prefix; re-run the crawler to update the schema
B Spark is using the schema from the first file read; enable mergeSchema in the Glue job
C Parquet doesn't support schema evolution; convert files to Avro format
D The IAM role for the Glue job doesn't have permission to read the new columns
B is correct. When reading multiple Parquet files with different schemas, Spark defaults to the schema of the first file it reads and silently drops extra columns. Setting spark.read.option("mergeSchema", "true") or using Glue's GlueContext.create_dynamic_frame.from_catalog() with additional_options={"mergeSchema": "true"} fixes this. Running the crawler updates the catalog schema but doesn't fix the Spark read behavior.
Q6. A financial services company stores transaction data in S3. Regulatory requirements mandate that 7-year-old data cannot be deleted by anyone, including AWS root account users. Which S3 feature should be used?
A S3 Versioning with MFA Delete enabled
B S3 Object Lock in Governance mode with a 7-year retention period
C S3 Object Lock in Compliance mode with a 7-year retention period
D S3 Lifecycle rule that prevents deletion for 7 years
C is correct. S3 Object Lock Compliance mode is immutable — not even the root account can delete objects before the retention period expires. Governance mode can be overridden by users with the s3:BypassGovernanceRetention IAM permission. S3 Versioning + MFA Delete protects against accidental deletion but doesn't prevent privileged deletions. S3 Lifecycle rules manage transitions/deletions, not WORM retention.
Q7. A Redshift table was recently used for a large DELETE operation that removed 40% of rows. Queries are now slower than before the DELETE. What should the data engineer do?
A Run ANALYZE to update query planner statistics
B Run VACUUM REINDEX to rebuild all sort keys
C Run VACUUM DELETE ONLY followed by ANALYZE
D Resize the Redshift cluster to add more nodes
C is correct. After a large DELETE, Redshift marks rows as "ghost rows" but doesn't reclaim storage immediately. VACUUM DELETE ONLY reclaims the space from deleted rows (faster than full VACUUM because it doesn't re-sort). Following with ANALYZE updates the planner statistics so it knows the table is now 40% smaller and chooses appropriate join strategies. Adding nodes doesn't fix row-level garbage collection.
Q8. A data team stores 3 years of clickstream data in S3 Standard. Data older than 30 days is rarely accessed and data older than 1 year is almost never accessed. The team wants to reduce storage costs while maintaining the ability to query any data within minutes. Which solution minimizes cost?
A Move all data immediately to S3 Glacier Deep Archive
B Set a lifecycle policy: Standard → Standard-IA at 30 days → Glacier Instant Retrieval at 365 days
C Move all data older than 30 days to S3 Glacier Flexible Retrieval
D Enable S3 Intelligent-Tiering on all objects
B is correct. The tiered lifecycle matches the access pattern: Standard for current data, Standard-IA (millisecond retrieval, lower cost) for data 30–365 days old, Glacier Instant Retrieval (millisecond retrieval, lowest cost) for data older than a year. Option A (Deep Archive) has 12–48 hour retrieval — fails the "within minutes" requirement. Option C (Glacier Flexible) has 1–12 hour retrieval — also fails. Option D (Intelligent-Tiering) works but costs more due to per-object monitoring fees.
