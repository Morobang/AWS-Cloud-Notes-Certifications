Introduction

The Data Engineering Pipeline

Domain 1 is the largest domain — it covers how data gets into AWS, how it gets transformed, and how those steps are automated and orchestrated.

The core data pipeline flow

Every data engineering system follows the same fundamental pattern: data is ingested from sources (databases, streams, files), transformed into a clean and useful format, and loaded into a destination for analytics or application use. Domain 1 covers all three phases: choosing the right ingestion service (Kinesis vs MSK vs DMS), picking the right transformation engine (Glue vs EMR vs Lambda), and orchestrating the steps together (Step Functions vs MWAA).

Key concepts to master

Batch ingestion
Process data in bulk at scheduled intervals. Lower cost, higher latency. Used for nightly ETL jobs.
Streaming ingestion
Process records continuously as they arrive. Higher complexity, lower latency. Used for real-time analytics.
ETL vs ELT
ETL: transform before loading (Glue). ELT: load raw first, transform in the warehouse (Redshift, Athena).
Medallion Architecture
Bronze (raw) → Silver (cleaned) → Gold (business-ready). Standard data lake layering pattern.
DPU
Glue Data Processing Unit: 4 vCPU + 16 GB RAM. The billing and sizing unit for Glue jobs.
Shard
Kinesis Data Streams unit of capacity: 1 MB/s write, 2 MB/s read, 1,000 records/s.
Task Statement 1.1

Perform Data Ingestion

Choosing the right service to move data from where it's generated to where it needs to be processed.

Streaming vs Batch: The fundamental choice

Before choosing a service, determine whether the data needs to be processed continuously (streaming) or in scheduled runs (batch). Streaming: fraud detection, real-time dashboards, IoT sensor alerts — data must be acted on within seconds. Batch: nightly sales reports, end-of-month analytics, data warehouse loads — hourly or daily is acceptable. Most production data lakes use both patterns: streaming for real-time operational data, batch for bulk historical loads.

📝

Kinesis Data Streams vs Firehose — the most tested distinction

Streams: you write a consumer; sub-second latency; replay possible; multiple consumers per shard. Firehose: fully managed delivery; 60–900 second buffering; no replay; one destination. If the scenario mentions <1 second latency or replay — it's Streams. If it says "load to S3 automatically with no code" — it's Firehose.

Kinesis Data Streams — Key Mechanics

🔀

Shards & Partition Keys

Each shard handles 1 MB/s writes and 2 MB/s reads. The partition key determines which shard a record goes to. High-cardinality keys (user ID, UUID) distribute traffic evenly. Low-cardinality keys (status = "active") create hot shards.

📊

IteratorAge Metric

GetRecords.IteratorAgeMilliseconds measures consumer lag. If it's growing, consumers can't keep up. Fix: add more consumer instances or split shards. This is the most important Kinesis monitoring metric.

⚡

Enhanced Fan-Out

Dedicated 2 MB/s read throughput per consumer per shard. Standard GetRecords shares 2 MB/s across all consumers. Use Enhanced Fan-Out when you have multiple consumers that all need full throughput.

🔄

Resharding

SplitShard doubles capacity; MergeShards reduces it. Cannot exceed the 500 shard limit per account per region without a service limit increase. Hot shard? Identify the hot partition key first, then split.

Task Statement 1.2

Transform and Process Data

Converting raw data into clean, structured, query-optimized formats using the right processing engine.

The transformation decision

The right transformation engine depends on three things: scale (megabytes vs terabytes vs petabytes), runtime (seconds vs minutes vs hours), and complexity (simple field renaming vs custom ML models). Lambda for lightweight event-driven; Glue for serverless Spark ETL; EMR when you need the full Hadoop ecosystem or custom libraries.

AWS Glue — Deep Dive

📚

Glue Data Catalog

Central metadata repository for all tables across your data lake. Stores schema, location, and partition information. Compatible with Hive Metastore. Used by Athena, EMR, Redshift Spectrum, and Glue jobs.

🕷️

Glue Crawlers

Automated schema discovery — point at S3, RDS, DynamoDB, etc. Crawler infers schema and creates/updates table definitions in the Data Catalog. Run on schedule or trigger on new data arrival.

⚙️

DPU Sizing

1 DPU = 4 vCPU + 16 GB RAM. G.1X = 1 DPU per worker; G.2X = 2 DPU per worker (32 GB RAM). Use G.2X to fix OOM errors. Add more workers for parallelism. Use Job Bookmarks to skip processed data.

🔁

DynamicFrames

Glue's abstraction over Spark DataFrames — handles inconsistent schemas with resolveChoice() and dropNullFields(). Convert to Spark DataFrame when standard Spark APIs are needed.

The single most impactful Athena optimization

Convert CSV to Parquet and partition by date: s3://lake/events/year=2024/month=01/day=15/. A 1 TB table queried for one day goes from scanning 1 TB ($5.00) to scanning ~2.7 GB ($0.01). That's a 500x cost reduction from two configuration changes.

Step Functions — Error Handling

🔄

Retry with Backoff

Configure MaxAttempts, IntervalSeconds, and BackoffRate on any Task state. Automatically retries transient failures (Glue job throttling, Lambda timeouts) with exponential backoff before failing.

🪤

Catch & Route

Define Catch to route failed tasks to error-handling states (send SNS alert, quarantine data, trigger manual review). States.ALL catches any error type.

🗺️

Map State

Process each item in an array in parallel — e.g., process 1,000 S3 files concurrently. Set MaxConcurrency to limit parallelism. Ideal for fan-out processing patterns.

⏳

Wait for Task Token

Pause workflow and wait for an external system to call back with a token. Used for human approval steps or long-running async processes. Standard workflow can wait up to 1 year.

💡

Choosing MWAA vs Step Functions

Step Functions: New AWS-native project, simple-to-medium pipeline, want visual workflow, lower cost for intermittent pipelines. MWAA: Existing Airflow DAGs, team knows Python/Airflow, need complex cross-team dependencies, want open-source portability. Cost matters: Step Functions charges per state transition (nearly free for once-daily pipelines); MWAA charges even when idle.

Exam Preparation

Practice Questions

Domain 1 questions — identify the answer before reading the explanation.

Q1. A company needs to ingest 500,000 clickstream events per second from their website for real-time fraud detection within 200 milliseconds. Which service should they use?
AAmazon Kinesis Data Firehose
BAmazon Kinesis Data Streams
CAmazon SQS
DAWS Glue Streaming
Answer: B — Kinesis Data Streams delivers sub-second latency to custom consumers. Firehose (A) buffers for 60–900 seconds — too slow for 200ms requirements. SQS (C) is a message queue, not optimized for this throughput/latency profile. Glue Streaming (D) adds processing latency.
Q2. A Glue ETL job reprocesses all 500 GB of S3 data every night, even though only ~2 GB of new files are added daily. What is the correct fix?
AIncrease the number of DPUs
BEnable Glue Job Bookmarks
CConvert data to Parquet format
DAdd pushdown predicates
Answer: B — Job Bookmarks track which S3 objects/rows have already been processed. Subsequent runs only process new data. A increases compute but still processes everything. C improves query performance. D reduces data read but requires explicit filter expressions — Bookmarks are the purpose-built solution.
Q3. IoT sensors send JSON records to Kinesis Data Streams. The team wants records stored in Amazon S3 as Parquet files automatically, with no custom consumer code. What is the BEST solution?
ALambda reads from Kinesis and writes Parquet to S3
BKinesis Data Firehose with format conversion enabled, using the Glue Data Catalog schema
CAWS Glue Streaming job reads from Kinesis and writes Parquet to S3
DAmazon MSK Connect with an S3 sink connector
Answer: B — Firehose with format conversion natively converts JSON to Parquet using the Glue Data Catalog schema — zero consumer code. Lambda (A) requires writing and maintaining code. Glue Streaming (C) works but requires managing a Glue job. MSK Connect (D) is for Kafka, not Kinesis Data Streams.
Q4. A Glue ETL job fails intermittently with "Java heap space" errors on large partitions. CloudWatch shows JVM heap at 96% before failure. What is the BEST immediate fix?
AUpgrade the worker type from G.1X to G.2X
BAdd more G.1X workers
CConvert input data from CSV to Parquet
DEnable Job Bookmarks
Answer: A — G.2X doubles memory per worker (32 GB vs 16 GB), directly addressing JVM OOM. B (more workers) adds parallelism but the same memory per worker — each worker still runs out of memory on large partitions. C and D address different problems (format efficiency and reprocessing).
Q5. A company migrates their on-premises Oracle database to Amazon Aurora PostgreSQL. They need minimal downtime and the migration must handle ongoing changes during the cutover window. What is the correct service combination?
AAWS DataSync for schema + data transfer
BAWS SCT for schema conversion + AWS DMS with Full Load + CDC mode
CAWS DMS alone for both schema conversion and data migration
DAWS Snowball for data transfer + manual schema creation
Answer: B — Oracle → Aurora PostgreSQL is a heterogeneous migration. SCT converts Oracle DDL/stored procedures to PostgreSQL syntax. DMS in Full Load + CDC mode migrates all existing data, then continues replicating changes — allowing a near-zero-downtime cutover. DMS alone (C) handles data but not schema conversion for heterogeneous migrations.
Q6. A data pipeline Step Functions state machine calls a Glue job. The Glue job occasionally fails due to transient throttling. How should the state machine be configured to handle this automatically?
AAdd a Wait state of 60 seconds before the Glue Task state
BAdd a Choice state after the Glue Task state to check for failure
CAdd a Retry configuration to the Glue Task state with MaxAttempts=3 and exponential backoff
DRun two Glue jobs in a Parallel state and use whichever succeeds first
Answer: C — Step Functions Retry on the Task state automatically re-attempts the Glue job on failure, with configurable interval and backoff. Wait (A) adds delay before execution, not after failure. Choice (B) requires manually re-entering the state — not automatic retry. Parallel (D) would run both jobs, doubling cost unnecessarily.
Q7. A team has 40 existing Apache Airflow DAGs running on-premises. They want to run them on AWS with minimal code changes. Which service is the BEST fit?
AAWS Step Functions
BAmazon MWAA (Managed Workflows for Apache Airflow)
CAWS Glue Workflows
DAmazon EventBridge Scheduler
Answer: B — MWAA runs Apache Airflow natively — existing DAG Python files run unchanged. Step Functions (A) is AWS-proprietary and would require rewriting every DAG. Glue Workflows (C) only orchestrates Glue components, not arbitrary tasks. EventBridge (D) provides scheduling but not full DAG orchestration.
Q8. An analytics team queries a 10 TB S3 table daily in Athena. Currently the table uses CSV format with no partitioning, costing ~$50/day. They want to reduce cost by at least 90%. What changes achieve this?
AEnable Athena result caching
BConvert to Parquet format with Snappy compression and add date-based partitioning
CSwitch from Athena to Amazon Redshift
DIncrease Athena workgroup scan limit
Answer: B — Parquet columnar format reduces scanned bytes by reading only needed columns (~10x reduction). Date partitioning limits scans to relevant date ranges (filters out irrelevant days). Combined, a typical daily query over one month's data goes from scanning 10 TB to scanning <100 GB — over 99% cost reduction. Result caching (A) only helps for exact duplicate queries.
