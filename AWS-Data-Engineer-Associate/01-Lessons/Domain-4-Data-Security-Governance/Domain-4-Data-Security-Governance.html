Introduction

Securing the Data Platform

Domain 4 covers the full security stack for AWS data engineering: who can access what, how data is protected at rest and in transit, and how to comply with data privacy regulations.

Security is layered — identity, network, encryption, compliance

A secure data platform has multiple defense layers: IAM and Lake Formation control who can see which tables and columns; VPC and network controls prevent data from leaving the private network; KMS encryption protects data at rest and in transit; and Macie, CloudTrail, and audit controls ensure compliance with privacy regulations like GDPR. The exam tests all four layers together.

The most common exam scenario: a data lake uses S3 + Glue Data Catalog. The question will ask which combination of IAM, Lake Formation, S3 bucket policy, and KMS provides the least-privilege access to specific tables or columns.

Key concepts to master

Least privilege
Grant only the permissions needed to perform a specific task. Default deny everything; explicitly allow what's required.
Envelope encryption
KMS generates a DEK (data encryption key) to encrypt data. The DEK itself is encrypted by a KMS CMK. Only the encrypted DEK is stored alongside the data.
Lake Formation
Fine-grained access control for the Glue Data Catalog. Column-level, row-level, and tag-based permissions on top of IAM.
SSE-KMS
Server-Side Encryption with Customer Managed Keys. S3 encrypts each object with a unique DEK, encrypted by your CMK. Enables key rotation and audit trail.
Cryptographic erasure
GDPR right-to-erasure implementation: delete the KMS key for a customer's data, making it permanently unreadable without deleting files.
Amazon Macie
ML-powered PII discovery in S3. Scans objects and classifies sensitive data (SSN, credit cards, emails). Findings route to EventBridge for automated remediation.
LF-TBAC
Lake Formation Tag-Based Access Control. Assign tags to tables/columns, grant permissions by tag. Scales better than per-resource grants for large data catalogs.
Data masking
Obscure sensitive data for non-privileged users: static (replace in copy), dynamic (mask at query time), tokenization, hashing, or nullification.
Task Statement 4.1

Apply Authentication and Authorization Controls

IAM policies, Lake Formation permissions, S3 bucket policies, VPC endpoints, and cross-account data sharing.

IAM + Lake Formation: two permission systems working together

When Lake Formation is enabled, access to Glue Data Catalog resources (databases, tables, columns) requires both IAM permissions AND Lake Formation permissions. IAM is the outer gate; Lake Formation is the inner gate. A user needs IAM permission to call the Glue API and Lake Formation permission to access the specific table. If either denies access, the request fails.

⚠️

Column exclusion vs column inclusion

Lake Formation column permissions use an exclusion model: you grant SELECT on a table and specify which columns to exclude, rather than listing which columns are allowed. This means if a new column is added to the table, it is automatically visible to the grantee unless explicitly excluded. Design grants with this in mind — exclude by sensitivity classification, not by column name where possible. Use LF-TBAC to tag columns as PII=true and exclude all PII-tagged columns.

S3 Bucket Policies for Data Lakes

🔐

Enforce HTTPS

Deny all S3 requests that don't use TLS. Condition: "aws:SecureTransport": "false" with Effect: Deny. Prevents data exposure over unencrypted connections. Apply to all data lake buckets as a baseline security control.

🏛️

VPC Restriction

Lock down a sensitive S3 bucket to only allow access from within a specific VPC. Condition: "aws:SourceVpc": "vpc-xxxx" with Effect: Deny on non-matching requests. Requires an S3 VPC Gateway Endpoint in the VPC.

🔗

S3 Access Points

Create named access points with per-use-case policies. Each data team gets their own access point (ARN) with a policy restricting them to their prefix. Simplifies bucket policy management — each access point policy is independent.

🌐

Cross-Account Access

Grant another AWS account access to S3 data: bucket policy allows the external account's principal, AND the external account's IAM policy allows access to the bucket. Both sides must grant permission. Use Resource Access Manager (RAM) for Lake Formation cross-account sharing.

How KMS envelope encryption works

When S3 encrypts an object with SSE-KMS: (1) S3 asks KMS to generate a unique Data Encryption Key (DEK). (2) KMS returns both the plaintext DEK and an encrypted copy of the DEK (encrypted with your CMK). (3) S3 uses the plaintext DEK to encrypt the object, then immediately discards the plaintext DEK. (4) The encrypted DEK is stored alongside the encrypted object in S3 metadata. To decrypt: S3 sends the encrypted DEK to KMS; KMS decrypts it using the CMK and returns the plaintext DEK; S3 uses it to decrypt the object. Your CMK never leaves KMS — only DEKs travel.

💡

KMS key types for the exam

AWS-owned keys: free, managed by AWS, no visibility or control — used by SSE-S3. AWS-managed keys: automatically created per-service (e.g., aws/s3), free, 3-year rotation, you can see them but can't control policy. Customer Managed Keys (CMK): you create and manage, $1/month + $0.03/10K API calls, custom key policy, manual or automatic annual rotation. CloudHSM keys: FIPS 140-2 Level 3, hardware-backed, highest compliance requirement.

Data Masking Strategies

🎭

Static Masking

Replace PII with fake-but-realistic values in a copy of the data. Original remains intact. Use for non-production environments (dev/test) so developers can work with realistic data without accessing real PII. AWS Glue DataBrew can detect and mask PII automatically.

⚡

Dynamic Masking (Redshift)

Mask data at query time based on the querying user's role — original data unchanged in storage. SQL: CREATE MASKING POLICY hide_ssn AS (val TEXT) RETURNS TEXT USING ('XXX-XX-XXXX'), then attach to column. Analysts see masked values; privileged users see real values.

🔢

Tokenization

Replace sensitive values with a random token that maps back to the original in a separate token vault. The application uses tokens; real values never leave the vault. Use for payment card data (PCI-DSS). Unlike hashing, tokenization is reversible for authorized lookups.

#️⃣

Hashing / Pseudonymization

Replace PII with a consistent hash (SHA-256 + salt). The same input always produces the same hash — allows joins across datasets without exposing real values. Not reversible without the original value. GDPR recognizes pseudonymization as a privacy-preserving technique (but not anonymization).

Task Statement 4.3

Prepare Logs for Audit and Comply with Data Privacy

GDPR compliance, Amazon Macie for PII discovery, CloudTrail for audit logging, and immutable storage with S3 Object Lock.

Privacy regulations and the data engineer's responsibilities

GDPR (EU) and similar regulations (CCPA, POPIA) impose specific technical requirements on how personal data is stored, accessed, and deleted. As a data engineer, your responsibilities include: enabling audit logging (who accessed what PII, when), implementing right to erasure (deleting or making unreadable all records for a specific individual), discovering where PII lives using Macie, and enforcing data retention and deletion using lifecycle rules and Object Lock.

Amazon Macie — Automated PII Discovery

🔍

What Macie does

Scans S3 buckets using ML to identify PII: names, emails, SSNs, credit card numbers, passport numbers, health information. Generates findings categorized by severity. Runs on a schedule or on-demand. Integrates with EventBridge for automated remediation.

🤖

Automated remediation

Macie finding → EventBridge rule → Lambda function → auto-block public access on the bucket OR move file to quarantine prefix OR notify security team via SNS. This pattern is heavily tested: the event-driven loop from finding to automated action.

📋

Custom data identifiers

Define regex patterns for organization-specific PII that Macie doesn't detect by default (e.g., internal employee IDs, account numbers in a proprietary format). Combine with managed identifiers for comprehensive coverage.

🗂️

Catalog PII metadata

After Macie discovery, tag Glue Data Catalog tables with PII classification metadata. Use these tags in Lake Formation LF-TBAC policies to automatically restrict access to any column or table tagged as PII=true, sensitivity=high, etc.

CloudTrail — Audit Logging

Two types of CloudTrail events for data engineers

Management events: control-plane API calls (create bucket, modify Glue job, update IAM policy). Enabled by default in all trails. Data events: data-plane operations (S3 GetObject/PutObject/DeleteObject, DynamoDB GetItem/PutItem, Lambda invocations). Disabled by default — must be explicitly enabled. For GDPR audit trails, you need S3 data events to prove who accessed specific PII files and when.

⚠️

Data events cost extra — enable selectively

CloudTrail data events cost $0.10 per 100,000 events. A high-throughput S3 data lake can generate millions of GetObject events per day — enabling data events on all buckets can cost thousands of dollars per month. Best practice: enable data events only on buckets that contain PII or regulated data. Use S3 prefix filtering to narrow further: only log events on s3://data-lake/silver/customers/, not the entire bucket.

✅

S3 Object Lock for immutable audit logs

CloudTrail logs must be tamper-proof for regulatory compliance. Store CloudTrail logs in a dedicated S3 bucket with Object Lock in Compliance mode. This prevents even the root account from deleting logs before the retention period expires. Typical retention: 1–7 years depending on regulation. Pair with S3 Log File Validation (CloudTrail feature) to detect if log files were modified or deleted after delivery.

Practice Questions

Domain 4 Exam Practice

These questions reflect the style and difficulty of actual DEA-C01 exam questions. The correct answer is highlighted in green.

Q1. A company has a data lake in S3. A data analyst needs to query a customer table in the Glue Data Catalog but must NOT see the ssn and credit_card columns. The analyst already has IAM permissions to call Athena and Glue APIs. What is the most straightforward way to enforce column-level restriction?
A Create a separate Glue table with those columns removed and grant IAM access to the new table
B Create an Athena view that excludes those columns and grant IAM access to the view only
C Use Lake Formation to grant the analyst SELECT on the customers table while excluding the ssn and credit_card columns
D Use an S3 bucket policy to deny GetObject on the Parquet files containing those columns
C is correct. Lake Formation provides native column-level security through the Grant Permissions dialog — you grant SELECT on the table and specify excluded columns. The analyst sees all other columns but ssn and credit_card are hidden. Option A (duplicate table) creates maintenance overhead. Option B (Athena view) works but Lake Formation is the purpose-built solution. Option D is not feasible — S3 objects contain all columns; you can't deny access to specific columns within a Parquet file via bucket policy.
Q2. A data engineering team stores sensitive customer data in S3. The security team requires that all data be encrypted using keys managed by the company, with a CloudTrail audit log of every time a data file is decrypted. Which S3 encryption option meets these requirements?
A SSE-S3 (AES-256) with S3 default encryption enabled
B SSE-C with customer-provided keys on each upload request
C SSE-KMS with a Customer Managed Key (CMK)
D Client-side encryption using the AWS Encryption SDK before uploading
C is correct. SSE-KMS with a CMK satisfies both requirements: (1) company-managed key — you create and control the CMK in KMS, including the key policy; (2) CloudTrail audit trail — every KMS Decrypt call (which happens on every S3 GetObject for SSE-KMS objects) is logged in CloudTrail with the caller's identity, timestamp, and context. SSE-S3 uses AWS-managed keys with no per-key audit. SSE-C doesn't use KMS, so no CloudTrail audit. Client-side encryption provides control but no automatic CloudTrail integration.
Q3. Amazon Macie discovers a finding that an S3 bucket containing customer PII has been made publicly accessible. The security team wants this to be automatically remediated within minutes without manual intervention. What is the most efficient architecture?
A Configure Macie to send email alerts to the security team who then manually block access
B Use AWS Config rules to check public access daily and send findings to an SQS queue
C Configure Macie to publish findings to Amazon EventBridge, then create an EventBridge rule that triggers a Lambda function to call s3:PutPublicAccessBlock on the affected bucket
D Enable AWS GuardDuty to detect and block public S3 access automatically
C is correct. Macie findings are published to EventBridge. Create an EventBridge rule that matches Macie findings of type "Policy:IAMUser/S3BucketPublic" and targets a Lambda function. The Lambda calls the S3 API to enable Block Public Access on the bucket. This is fully automated, event-driven, and responds within seconds. Option A is manual. Option B (AWS Config) runs checks on a schedule, not in real-time. GuardDuty doesn't take automated remediation actions — it detects threats but doesn't block them.
Q4. A company must comply with GDPR. A customer submits a "right to erasure" request. Their data exists in: (1) a Redshift data warehouse, (2) an S3 data lake with Parquet files, and (3) a DynamoDB table. What is the correct approach for each?
A Delete the entire tables from all three services and restore from backup after removing the customer's rows
B DELETE the customer's row in Redshift, use Iceberg row-level DELETE or file rewrite in S3, and DeleteItem in DynamoDB
C Encrypt the customer's data with a new key and delete the key in all three services
D Set the customer's PII fields to NULL in all three services without deleting the rows
B is correct. Each service has the appropriate targeted deletion mechanism: Redshift supports standard SQL DELETE by customer_id; S3 data in Iceberg format supports row-level DELETE, or otherwise requires rewriting the Parquet files excluding that customer; DynamoDB's DeleteItem removes the item by its partition key. Option C (cryptographic erasure) is a valid alternative for S3, but not the simplest for Redshift and DynamoDB which have native delete operations. Option D (nullification) is pseudonymization, not erasure — it may not satisfy GDPR right to erasure if indirect identifiers remain.
Q5. A data analyst needs to query data in a Glue Data Catalog table that is governed by Lake Formation. The analyst has an IAM policy with glue:GetTable and athena:StartQueryExecution permissions, but queries still fail with an "access denied" error. What is the most likely cause?
A The analyst's IAM role needs s3:GetObject permission on the underlying S3 bucket
B The analyst has not been granted SELECT permission on the table in Lake Formation
C Athena workgroup permissions are blocking the query
D The Glue crawler needs to be re-run to refresh the table metadata
B is correct. When Lake Formation is enabled, both IAM and Lake Formation permissions are required. The analyst has IAM permissions to call the APIs but hasn't been granted data-level access in Lake Formation. Lake Formation acts as an additional authorization layer on top of IAM — IAM permits the API call, Lake Formation permits the data access. Even with full IAM access, without a Lake Formation GRANT, the query fails. Option A is also required but is typically handled through the Lake Formation service role mechanism, not direct bucket permissions.
Q6. A Redshift data warehouse serves both a privileged finance team (who need to see full SSNs) and an analyst team (who should see masked SSNs like 'XXX-XX-XXXX'). The same underlying table must serve both use cases without maintaining two copies of the data. What Redshift feature achieves this?
A Row-level security (RLS) policy filtering rows by user role
B Lake Formation column exclusions for the analyst role
C Dynamic data masking policy attached to the SSN column, conditioned on user role
D Two separate Redshift views: one with SSN for finance, one with NULL SSN for analysts
C is correct. Redshift's Dynamic Data Masking (DDM) masks column values at query time based on the role of the querying user. The original data is unchanged in storage. Finance users assigned to the privileged role see the real SSN; analysts see 'XXX-XX-XXXX'. The masking policy is attached to the column and checks current_user or role membership. Option D (two views) works but requires ongoing maintenance. RLS filters rows, not column values. Lake Formation column exclusions hide the column entirely (null, not masked).
Q7. A compliance officer needs to prove that no one accessed a specific S3 bucket containing financial records over the past 90 days. Which AWS service and configuration provides this evidence?
A S3 Server Access Logs enabled on the bucket
B AWS Config rules monitoring the bucket's configuration
C AWS CloudTrail with S3 data events enabled for the specific bucket
D Amazon GuardDuty S3 protection enabled on the account
C is correct. CloudTrail data events for S3 log every GetObject, PutObject, DeleteObject call — capturing who (IAM principal), when (timestamp), and from where (IP, VPC). This provides a complete access audit trail for the compliance officer. S3 Server Access Logs (A) also capture access but are less reliable (best-effort delivery, less structured) and lack the depth of CloudTrail. AWS Config (B) tracks configuration changes, not data access. GuardDuty (D) detects threats but doesn't provide a comprehensive access log for compliance purposes.
Q8. A company needs to ensure their AWS CloudTrail audit logs stored in S3 cannot be deleted for 5 years, even by administrators with full IAM access. What is the correct configuration?
A Enable S3 Versioning and MFA Delete on the CloudTrail bucket
B Create an S3 bucket policy denying DeleteObject for all principals except the root account
C Enable S3 Object Lock in Compliance mode with a 5-year retention period on the CloudTrail bucket
D Enable S3 Object Lock in Governance mode with a 5-year retention period on the CloudTrail bucket
C is correct. S3 Object Lock Compliance mode is the only option where no one — not even the AWS root account — can delete objects before the retention period expires. Governance mode (D) can be bypassed by any IAM principal with the s3:BypassGovernanceRetention permission, including administrators. MFA Delete (A) adds a factor for deletion but doesn't prevent administrators with MFA from deleting. Bucket policies (B) can be modified by bucket owners with sufficient IAM permissions.
